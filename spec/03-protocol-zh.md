# 03 — 协议与方法论

本文档定义 LongHorizonMemSysBench v1 的实验协议和统计方法。它是如何运行、比较和分析 episode 的规范参考。所有实现代码（智能体 harness、实验运行器、统计分析）均实现本文声明的规则。

## 1. 反事实回放协议

### 1.1 固定外生世界

核心实验设计是**反事实回放**：每个 episode 在所有记忆条件下完全相同地运行，仅智能体附带的记忆系统不同。在 v1 中，证据世界是*固定和外生的*。

每个 episode 由 `(episode_id, seed)` 对定义。从该对出发，模拟器核心确定性地生成：

1. 有序的**世界事件**序列（`inject`、`change`、`retract`），定义智能体感知到的内容。
2. 对齐的**探针**序列（`task-completion`、`drift-check`、`retrieval`），带有从每个探针步骤的世界状态派生的黄金答案（已揭示减去已撤回的事实）。
3. **world_event_hash**：基于有序事件+探针计划计算的稳定、确定性哈希值。

world_event_hash 在每个 `(episode_id, seed)` 上跨所有条件**完全相同**。这是完整性保证：每个记忆系统面临完全相同的证据。如果在相同 `(episode_id, seed)` 的条件之间检测到哈希不匹配，该运行无效，实验中止。

### 1.2 智能体动作不改变世界（v1）

在 v1 中，智能体动作仅是*观察性的*：智能体感知证据，查询其记忆，并回答探针，但其动作不改变 `WorldState`。世界事件计划完全外生，无论智能体做什么都相同地进行。这是 v1 的刻意简化：

- 消除由于智能体行为分歧改变后续探针可用证据而带来的混淆（智能体 A 做出不同决定，改变未来世界，使反事实比较不可能）。
- 实现跨条件的干净、配对统计比较。
- 保持模拟器确定性和可复现性。

交互式世界（智能体动作改变未来证据）推迟到 v2。

### 1.3 对齐探针点

探针在 episode 计划的固定步骤插入。同一个探针（相同的 `probe_id`、`kind`、`query`、`gold`）在给定 episode 的每次条件运行中以相同的步骤出现。这使得比较在探针级别是**配对**的。

探针黄金答案从探针步骤的 `WorldState` 程序化派生。具体来说：
- 已被 `inject` 且尚未 `retract` 的事实 → 出现在黄金答案中。
- 已被 `retract` 的事实 → 在黄金答案中不存在（即使智能体的记忆系统仍然持有它）。
- 已被 `change` 的事实 → 仅最新版本在黄金答案中。

丰富的计划（事件 + 探针 + 黄金答案）作为数据集的一部分被冻结并校验和。

### 1.4 会话边界

Episode 被划分为**会话（sessions）**，模拟长时间跨度任务中的时间间隔。会话边界是确定性的（从种子派生）且在所有条件间相同。在会话边界：

- 智能体的工作上下文被清除（上下文窗口重置）。
- 记忆系统持久存在（对所有条件，除 `no_memory` 外）。
- 下一个会话以计划中的下一个世界事件开始。

这是迫使智能体依赖其记忆系统的机制：早期会话的信息只有在记忆系统已存储且能检索时才能访问。

## 2. 无记忆对照组策略

### 2.1 定义

**无记忆对照组**（no-memory control）是一种智能体没有持久记忆系统的实验条件。它是所有记忆系统比较的反事实基线。

无记忆对照组在会话之间是**无状态的**。在每个会话边界，智能体的整个状态被丢弃。智能体在会话中唯一可用的信息是：

1. 该会话期间揭示的事件（当前会话上下文）。
2. 固定的上下文预算 **B**——与所有其他条件为当前会话上下文提供的相同工作集大小。

无记忆对照组不保留任何来自先前会话的信息。它不能查找过去的事件、过去的智能体推理或过去的探针答案。这是反事实的定义：没有记忆，可能的性能是什么？

### 2.2 上下文预算 B

每个条件，包括无记忆，接收相同的工作集上下文预算 **B**（当前会话事件 + 智能体推理）。预算以 token 为单位并在所有条件间完全相同地强制执行。这确保记忆条件和无记忆条件之间的任何性能差异可归因于记忆系统提供*超出上下文窗口*的信息的能力，而非更大的工作集。

### 2.3 ROI 反事实角色

无记忆条件在 Memory ROI 公式中提供基线 `score(no_memory)`：

```
gain = score(system) − score(no_memory)
normalized_gain = clamp(gain / max(ε, max_achievable − score(no_memory)), −1, 1)
ROI = mean(normalized_gain) / mean(normalized_cost)
```

没有无记忆反事实，不可能区分真正改善性能的记忆系统和仅仅与更容易的 episode 相关的系统。

无记忆条件本身的 ROI 是 `N/A`（非零），因为 ROI 定义为相对于基线的比较，基线不能与自身比较。这在所有报告中被强制执行，绝不显示为 `0` 或被省略。

## 3. 赛道策略：原生和受控

### 3.1 原生赛道（主要）

**原生赛道**是主要排行榜。每个记忆系统以其默认配置部署：

- 自己的内部 LLM（如果有），使用自己的模型、提示和 token 使用量。
- 自己的索引、嵌入、检索和存储策略。
- 自己的分块、top-k、相似度阈值等默认值。

所有内部资源消耗（LLM token、嵌入调用、存储字节、检索延迟）都被仪表化并计入系统成本向量。

原生赛道回答部署问题：*"如果我按原样安装这个记忆系统，我的 Memory ROI 是多少？"*

### 3.2 受控赛道（次要）

**受控赛道**是次要的子研究，与原生赛道分开报告，绝不合并到主要排行榜中。

在受控赛道中，如果记忆系统支持固定其内部模型，则所有系统使用相同的模型。这隔离了记忆*架构*（事实如何存储、组织和检索）的效果与内部模型*质量*的效果。内部 LLM 较弱的记忆系统可能在原生赛道上表现不佳，但在受控赛道上匹配或超越，揭示架构本身是否健全。

受控赛道是**可选的**：不支持模型固定的系统在受控赛道上不存在。它始终是次要分析；结论从原生赛道得出。

### 3.3 分离规则（绝不混合）

原生赛道和受控赛道的结果在任何单一排行榜、表格或图表中**绝不混合**。它们在记分卡的不同章节中呈现。任何聚合、排名或统计比较仅在一个赛道内进行。违反此规则会混淆两个不同的问题（系统部署状态 vs. 仅架构）并产生无法解释的结果。

## 4. 统计方法

### 4.1 实验设计

配对反事实设计：

- **每个任务族 N 个 episode**（pilot 默认约 20 个每组，可配置）。
- **K 个种子**（pilot 默认 ≥3，可配置），每个产生不同的 episode 实例。
- 每个 (episode_id, seed) 在每个条件下运行，产生每个条件、每个 episode 的配对观察。

分析单位是 episode-条件对。指标首先按 episode 聚合（在该 episode 内的探针上），然后按条件聚合（在 episode 上），然后两两比较（条件 vs. 无记忆）。

### 4.2 Bootstrap 置信区间

所有核心指标（Memory ROI、任务分数、drift_index、检索指标）均以 **bootstrap 置信区间**（默认 95% CI，≥10,000 次重采样，可配置）报告。

步骤：
1. 从 episode 池中有放回地抽样 N 个 episode。
2. 在重采样集上计算指标。
3. 重复 R 次。
4. 将第 2.5 和 97.5 百分位数报告为 95% CI。

Bootstrap CI 优于参数区间，因为比率（如 ROI）的抽样分布和 episode 难度分布不保证是正态的。

### 4.3 效应量

除 CI 外，还为主要比较（每个条件 vs. 无记忆）报告标准化**效应量**。pilot 设计有能力检测预先声明的最小效应量。效应量指标和最小值在运行配置中声明；默认是每 episode 分数增量的 Cohen's d。

### 4.4 包含失败运行（不丢弃）

失败（超时、崩溃、畸形输出）的运行不从分析中丢弃。失败评分如下：

- 该 episode 的任务分数 = 0（智能体未能完成任务）。
- 成本向量 = 失败点之前的实际已发生成本（token 已消耗，API 调用已完成，崩溃前存储已写入）。
- 在 episode 结果中记录失败标记以供诊断报告。

丢弃失败运行会偏向上调高结果（在困难 episode 上失败的系统看起来更强）。包含它们确保结果对最坏情况行为诚实。部分成功（在失败前回答了一些探针）包括已回答的探针正常评分和未回答的探针评分为 0。

### 4.5 帕累托分析

单独的 Memory ROI 可能隐藏权衡。高 ROI 但低绝对性能的条件可能没有用；低 ROI 但最高绝对分数的条件可能在某些应用中被优先选择。因此，每个记分卡包含一个**帕累托前沿**图：

- X 轴：归一化成本（每 episode 均值）。
- Y 轴：归一化增益（每 episode 均值）。
- 每个条件是一个点；帕累托前沿连接非支配条件。
- 无记忆对照组按定义绘制在 (0, 0)（其自身的 ROI 是 N/A，但它在帕累托平面上作为参考点）。

帕累托图附带增益下限和成本上限线，以便观察者识别实际关心的区域。绝不仅仅报告没有不确定性区间和帕累托上下文的裸排行榜数字；这是报告模块强制执行的结构性要求。

## 5. 失败策略

### 5.1 超时和重试

每个 episode 运行有一个预先声明的**超时**（挂钟时间，按任务族可配置）。如果智能体未能在超时内完成所有会话，运行终止并评分为失败。

每个操作（LLM 调用、记忆适配器调用）有有限的**重试**次数（默认 2，可配置）。如果所有重试耗尽，运行评分为失败。重试计入成本向量。

### 5.2 崩溃策略

如果智能体进程、记忆系统或 harness 崩溃：
- 运行评分为**任务失败**（剩余探针分数 = 0）。
- 崩溃前发生的所有成本**保留**在成本向量中（token 已收费，API 调用已计数，存储已测量）。
- 崩溃记录与堆栈跟踪和诊断信息一起记录。

### 5.3 畸形输出

如果智能体产生无法被程序化检查器解析的输出：
- 探针评分为 0。
- 畸形输出被记录。
- 不尝试重试（避免在无界的解析重试循环上膨胀成本）。

### 5.4 不支持的适配器操作

记忆系统的功能集各不相同。有些可能不支持更新、删除或反思。当适配器收到它不支持的操作调用时：

- 必须抛出明确定义的 `UnsupportedOperation`（不静默崩溃）。
- Harness 捕获此异常，记录，并用剩余操作继续。
- 能力差距记录在条件的结果中（因此报告可以注明"系统 X 不支持更新，这可能影响漂移分数"）。

优雅的能力降级优于崩溃整个运行。然而，如果缺失的能力使探针无法回答（例如漂移探针需要检查撤回的事实是否仍可检索），该探针对该条件评分为 0。

### 5.5 畸形记忆状态

如果记忆系统进入不一致状态（损坏的索引、静默检索失败、重复 ID）：
- 条件标记为诊断标记。
- 运行继续（避免因丢弃困难情况而偏向上调结果）。
- 事件在记分卡的诊断附录中报告。

### 5.6 排除规则

唯一完全从分析中排除 episode 运行的情况是同一 `(episode_id, seed)` 的条件之间存在**world_event_hash 不匹配**。这表明模拟器或数据集存在确定性 bug，而非记忆系统属性，该运行是无效数据。此类排除必须罕见（正常操作中为零）并醒目地报告。
