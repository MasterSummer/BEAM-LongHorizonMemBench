## 1. Memory ROI — Headline Metric

Memory ROI answers: *did the memory system improve performance, and at what cost?*
It is a counterfactual ratio computed per system per family per track, never a single
bare leaderboard number.

### 1.1 Definitions

For each episode `e` and seed `s`, under condition (system) `c`:

```
score(c, e, s)    = task score ∈ [0, max_achievable(e)]   (see Section 4)
gain(c, e, s)     = score(c, e, s) − score(no_memory, e, s)
```

The gain floor is `max(0, gain)`. A negative gain is retained for ROI computation
(producing negative ROI) but the gain floor is also reported to distinguish
"not harmful" from "truly beneficial" systems.

**Normalized gain** bounds the counterfactual improvement to [-1, 1], preventing
any single episode from dominating the aggregate because of an atypically large gain:

```
normalized_gain(c, e, s) = clamp(
    gain(c, e, s) / max(ε, max_achievable(e) − score(no_memory, e, s)),
    −1,
    1
)
```

where `ε = 0.001` (prevents division by zero when no_memory already achieves the
maximum). `max_achievable(e)` is the known perfect score for the episode, determined
by the task family's ground truth.

**Cost vector** → **scalar cost**: each episode-condition run produces a `CostVector`
(see Section 1.3). The vector is scalarized to **tokens-equivalent** via a declared
conversion sheet (`configs/cost_weights.yaml`):

```
cost(c, e, s) = scalarize(CostVector(c, e, s))
```

**Memory ROI** aggregates over episodes and seeds:

```
mean_gain(c) = mean_{e,s} normalized_gain(c, e, s)
mean_cost(c) = mean_{e,s} cost(c, e, s)

ROI(c) = mean_gain(c) / mean_cost(c)
```

ROI is reported with **bootstrap CI** (95%, seeded, n=10,000 resamples) over the
paired (episode, seed) units. Units: gain per token-equivalent cost (typically
reported per 1,000 tokens for readability).

### 1.2 Edge-Case Policies

| Case | Policy |
|---|---|
| **No-memory control** | ROI = `N/A` (not 0, not reported). The no-memory condition IS the baseline used to compute gain for all other systems; computing ROI for it is undefined by construction. |
| **Near-zero cost** (`mean_cost < ε_cost`) | ROI is flagged `undefined-lowcost`, **never** `+inf` or `NaN`. The Pareto front and cost-vector breakdown are still reported; the gain is reported separately. `ε_cost = 1.0` (less than 1 token-equivalent per episode on average). |
| **Negative gain** (`gain < 0`) | Produces negative ROI. This is allowed and reported — it means the memory system harmed performance. Always paired with the gain floor (which floors at 0) and Pareto context so a negative-ROI system is not misread as "better than baseline." |
| **max_achievable ≈ score(no_memory)** | The `ε` clamp in `normalized_gain` prevents division by zero. A fully saturated baseline (no room for improvement) produces near-zero normalized gain regardless of the raw gain — reflecting that the task doesn't test memory. Such episodes are flagged for the analyst. |
| **Undefined conditions** | Episodes where `no_memory` timed out or crashed are excluded from pairwise gain computation for that (episode, seed); the system's raw score + cost are still reported. Documented as a partial pair. |
| **NaN / +inf**  | Forbidden. Any computation path that would produce `+inf` or `NaN` must instead emit `undefined-lowcost` or `N/A`. |

### 1.3 CostVector

Each `CostVector` records the full lifecycle cost of running the episode under a
given condition. Fields (all cumulative over the episode):

| Field | Unit | Description |
|---|---|---|
| `agent_input_tokens` | tokens | Tokens fed into the agent LLM (prompt + context). |
| `agent_output_tokens` | tokens | Tokens generated by the agent LLM. |
| `mem_internal_in_tokens` | tokens | Tokens consumed by the memory system's internal LLM calls (e.g., Mem0 extraction, Cognee cognify, Letta block edits). Counted inside `memory_scope()`. |
| `mem_internal_out_tokens` | tokens | Tokens generated by the memory system's internal LLM. |
| `embedding_tokens` | tokens | Total tokens embedded (input-side) for vector storage. |
| `embedding_calls` | count | Number of embedding API calls. |
| `storage_bytes` | bytes | Peak or cumulative storage used by the memory backend. |
| `retrieval_latency_ms` | ms | Total wall-clock latency of `search()` calls. |
| `write_latency_ms` | ms | Total wall-clock latency of `add_memory()` calls. |
| `update_latency_ms` | ms | Total wall-clock latency of `update_memory()` / `delete_memory()` calls. |
| `reflection_tokens` | tokens | Tokens consumed by explicit `reflect()` / consolidation calls (counted under `memory_scope()`). |
| `num_retrieval_calls` | count | Number of `search()` invocations. |

**Exclusions**: one-time dataset-generation tokens, judge tokens, and surface-rendering
tokens are NOT counted in the system CostVector. Each runs under its own scope.

**Scalarization**: `scalarize(cv, weights, sheet) -> float` produces a single
tokens-equivalent number. Latency and storage are converted using the conversion sheet
in `configs/cost_weights.yaml` (e.g., 1 ms latency = 0.1 token-equivalent; 1 KB storage
= 0.01 token-equivalent). The scalarization weights are fixable per run-config and
documented in every scorecard.

### 1.4 Worked Example

Consider a 3-episode family with `max_achievable = 1.0` for all episodes.

| Episode | no_memory score | System A score | max_achievable − no_memory | gain | normalized_gain | System A cost (tokens-equiv) |
|---|---|---|---|---|---|---|
| e1 | 0.30 | 0.70 | 0.70 | +0.40 | clamp(0.40/0.70, −1, 1) = 0.571 | 5200 |
| e2 | 0.45 | 0.60 | 0.55 | +0.15 | clamp(0.15/0.55, −1, 1) = 0.273 | 4800 |
| e3 | 0.20 | 0.90 | 0.80 | +0.70 | clamp(0.70/0.80, −1, 1) = 0.875 | 5100 |

```
mean_gain(A)   = (0.571 + 0.273 + 0.875) / 3 = 0.573
mean_cost(A)   = (5200 + 4800 + 5100) / 3 = 5033.3
ROI(A)         = 0.573 / 5033.3 = 1.14 × 10⁻⁴ per token
               ≈ 114 per million tokens-equivalent
```

A bootstrap CI (95%) over the 3 paired differences might yield `[92, 136]` per
million tokens. The gain floor for System A is `mean(max(0, gain)) / mean(cost)`,
which in this case equals the ROI (all gains positive).

**Negative-gain example**: System B with mean_normalized_gain = −0.15, mean_cost = 4000:

```
ROI(B) = −0.15 / 4000 = −3.75 × 10⁻⁵ per token
       ≈ −37.5 per million tokens-equivalent
```

Reported as "negative ROI (system harmed performance)" with gain floor = 0.

**Near-zero cost example**: System C with mean_cost = 0.5 (< ε_cost = 1.0):

```
ROI(C) = undefined-lowcost (NOT +inf)
```

The gain (if any) is reported separately; the cost-vector breakdown is preserved.

## 2. Goal Drift & Behavioral Stability (Dim 3)

Goal drift measures whether a memory system helps the agent maintain behavioral
consistency over a long horizon. Drift is measured via **programmatic invariants**,
not an LLM rubric. A sparse judge (see `spec/01-overview.md`) handles only the
small fraction of cases that are not programmatically decidable.

### 2.1 Definitions

Each episode carries a set of **invariants**: active objectives and constraints
with validity windows `[t_start, t_end)` and explicit supersession events.
A family module provides these — see `spec/04-datasets.md`.

**Three violation categories** (each IS drift):

1. **Stale-fact use** (drift — category A): the agent uses or cites a fact `F`
   after step `t > F.retracted_at`. The fact was once valid but has been retracted
   or superseded.

2. **Constraint violation** (drift — category B): the agent violates a still-active
   constraint `C` at step `t` where `t ∈ C.validity_window` AND no superseding event
   has lifted `C`.

3. **Behavioral flip** (drift — category C): the agent reverses a prior decision
   or stated position at step `t` without a triggering world event (inject, change,
   retract) between the reversal and the prior statement.

**NOT drift**: changing a conclusion, behavior, or stated fact AFTER a valid
superseding or retraction world event. This is correct adaptation and must NOT
be penalized.

### 2.2 Formula

For each episode, count violations over the set of aligned probes `P`:

```
drift_count(category, e, s) = number of aligned probes where the agent's answer
                               commits a violation of that category

drift_weighted(e, s) = w_A × drift_count(A) + w_B × drift_count(B) + w_C × drift_count(C)

drift_index(c, e, s) = drift_weighted(e, s) / (|P_A| × w_A + |P_B| × w_B + |P_C| × w_C)
```

where `|P_X|` is the number of probes of category X, and default weights are
`w_A = 1.0, w_B = 1.5, w_C = 1.0`. Weights are configurable; `w_B` is higher
because constraint violations are the most severe form of drift (active rule
breaking without permission).

`drift_index ∈ [0, 1]`, where 0 = no drift (perfect stability), 1 = drifted on
every aligned probe.

The per-episode drift report also emits:
- Per-category violation counts
- Offending probe IDs
- Judge-fallback share (fraction of probes where the judge was needed because
  programmatic detection was infeasible)

### 2.3 Edge-Case Policies

| Case | Policy |
|---|---|
| **Valid adaptation** (event → change) | NOT counted as drift. The drift detector must verify: did a world event occur AFTER the agent's previous position and BEFORE the changed position? If yes → not drift. |
| **Ambiguous probe** (programmatic checker cannot determine) | Flagged as `judge-needed`. The sparse judge evaluates it; the judge share is reported separately and bounded (≤ 20% of drift probes). If the judge share exceeds the bound, the episode is flagged for manual audit. |
| **No aligned probes** | drift_index = `N/A` (no probes measure drift in this episode). |
| **Retraction chain** (F retracted at t1, re-injected at t2) | The fact is valid from t2 onward. Using it at t3 > t2 is NOT stale-fact use. The drift detector tracks validity windows, not binary retracted/not. |
| **Behavioral flip with implicit event** | If the agent's change is EXPLICITLY justified by an earlier event (even if the justification is delayed), the checker should accept it as valid adaptation. Only UNJUSTIFIED reversals count as drift. |

### 2.4 Worked Example

Episode with 10 probes: 4 stale-fact (category A), 3 constraint (B), 3 behavioral (C).

**Scenario 1 — clean run (no drift):**

| Probe | Category | Agent action | World context | Verdict |
|---|---|---|---|---|
| p1 | A | Uses fact F at step 2 | F valid until step 5 | No violation |
| p2 | A | Uses fact G at step 6 | G retracted at step 3 | **DRIFT — stale-fact use** |
| p3 | B | Follows constraint C at step 4 | C active, no supersession | No violation |
| p4 | B | Breaks constraint C at step 8 | C lifted by event at step 7 | **NOT drift** (valid adaptation) |
| p5 | C | Reverses decision at step 5 | No event between prior and step 5 | **DRIFT — behavioral flip** |

Assume all other probes (p6-p10) pass without violation. Results:
- Stale-fact violations: 1 (p2)
- Constraint violations: 0
- Behavioral flips: 1 (p5)

```
drift_weighted = 1.0 × 1 + 1.5 × 0 + 1.0 × 1 = 2.0
denominator    = 4 × 1.0 + 3 × 1.5 + 3 × 1.0 = 4.0 + 4.5 + 3.0 = 11.5
drift_index    = 2.0 / 11.5 = 0.174
```

**Scenario 2 — valid adaptation not penalized:**

Same episode. The agent obeys constraint C at step 3. At step 7, an event lifts C.
At step 9, the agent acts contrary to the former C. The drift checker verifies:
event at step 7 occurred BEFORE the change at step 9 → valid adaptation → NOT drift.

```
drift_index = 0.0  (no actual violations)
```

This is the critical false-positive guard: the agent correctly adapted to a world
change and must not be penalized.

## 3. Retrieval Quality (Dim 4)

Retrieval quality measures how well the memory system returns relevant items,
deconfounded from the agent's query quality.

### 3.1 Definitions

Two measurement modes, reported separately:

**Endogenous retrieval**: the agent's own `search()` calls during the episode.
Scored against the known relevant `fact_id` set at each probe step.
*Weak agent + strong memory* will show low endogenous p@k because the agent
issued poor queries — but this is an agent problem, not a memory problem.

**Oracle retrieval**: a fixed, agent-independent set of benchmark queries with
known relevant memory IDs, fired directly at `adapter.search()`. This isolates
the memory system's retrieval quality.

### 3.2 Formulas

For a single query `q` with gold relevant set `R_q` (size `|R_q|`) and returned
top-k results `S_{q,k}`:

```
precision@k(q) = |S_{q,k} ∩ R_q| / k
recall@k(q)    = |S_{q,k} ∩ R_q| / |R_q|
```

For a set of queries `Q`:

```
mean_precision@k = (1/|Q|) × Σ_{q∈Q} precision@k(q)
mean_recall@k    = (1/|Q|) × Σ_{q∈Q} recall@k(q)
```

**context_relevance**: fraction of returned items that are valid (not retracted)
at the probe step. Calculated as `|{i ∈ S_{q,k} : i.fact is current}| / k`.

Default `k = 10`. Configurable per run.

Reported as a triple per mode: `(p@k, recall@k, context_relevance)`.

### 3.3 Edge-Case Policies

| Case | Policy |
|---|---|
| **Empty gold set** (`|R_q| = 0`) | recall@k = N/A; precision@k = 0 always. Flagged as a non-retrieval probe. |
| **Empty result set** (`|S| = 0`) | p@k = 0, recall@k = 0. |
| **Fewer than k results** | Denominator for p@k is still `k` (the *requested* k), not the returned count. This penalizes systems that return too few results. |
| **Duplicate results** | Deduplicated before intersection (each `memory_id` counted once). |
| **No oracle probes defined** | Oracle retrieval = N/A for the episode; endogenous only. |
| **No retrieval calls in episode** | Both modes = N/A. Indicates the agent never used memory search (relevant for utilization analysis). |

### 3.4 Worked Example

Gold relevant set for an oracle probe: `R = {m1, m2, m3, m4}`, `|R| = 4`.
System returns top-10: `S = {m1, m5, m2, m6, m7, m8, m9, m10, m11, m12}`.
`k = 10`.

```
|S ∩ R| = 2   (m1, m2)
precision@10 = 2 / 10 = 0.20
recall@10    = 2 / 4  = 0.50
```

If `m8` was retracted at this probe step, `context_relevance = 9/10 = 0.90`
(1 invalid result out of 10).

**Endogenous vs oracle split example**:

| Mode | System | p@10 | recall@10 | Interpretation |
|---|---|---|---|---|
| Endogenous | Weak agent + FakePerfect | 0.15 | 0.12 | Agent queried poorly |
| Oracle | FakePerfect (same run) | 0.98 | 0.95 | Memory system retrieves perfectly |

The split confirms the memory system is strong but the agent is the bottleneck —
preventing a false conclusion that "the memory system failed."

## 4. Task Performance & Goal-Directed Utilization (Dim 2)

### 4.1 Definitions

**Task score** (`task_score`): the fraction of episode probes answered correctly,
aggregated by the family `Checker` (see `spec/04-datasets.md`). Where a probe is
open-ended (not programmatically checkable), the sparse judge provides a score;
judge contribution is reported separately and bounded.

```
task_score(c, e, s) = ( Σ deterministic_probe_scores + Σ judge_probe_scores ) / |probes|
```

`task_score ∈ [0, 1]`. `max_achievable(e) = 1.0` when all probes carry a known
gold answer; for episodes with judge-only probes it may be less (declared per
episode).

**Utilization rate** (`utilization_rate`): of the probes whose correct answer
REQUIRES a fact learned in an earlier session (cross-session dependent), the
fraction the agent answered correctly. Probes whose required facts are available
in the current session's context are excluded — they don't test memory.

```
utilization_rate(c, e, s) =
    |{p ∈ cross_session_probes : p.correct}| / |cross_session_probes|
```

where `cross_session_probes` is the subset of probes where the gold answer
depends on ≥1 fact from a session prior to the probe's session.

**Improvement over time**: optional per-episode trend analysis — does the agent's
task score increase across sessions as more memory accumulates? Reported as a
supplementary chart, not a headline number.

### 4.2 Edge-Case Policies

| Case | Policy |
|---|---|
| **No cross-session probes** | utilization_rate = N/A. All probes were answerable from context. |
| **Judge-only probe** | The judge provides a `[0,1]` score + rationale. Judge scores are tagged and reported in a separate `judge_contribution` field. The judge weight in any composite is capped at 20%. |
| **Timeout / crash** | Probe answered = 0 (task failure). The run is included, not dropped. |
| **Agent answer is empty** | Scored as incorrect (0). Not treated as missing data. |
| **Probe answer format mismatch** | If the agent's answer cannot be parsed by the checker, the probe is scored as 0 and logged as `format_error`. The episode continues. |

### 4.3 Worked Example

Episode with 6 probes across 3 sessions:

| Probe | Session | Cross-session? | Agent correct? | Score |
|---|---|---|---|---|
| p1 | s1 | No (fact in context) | Yes | 1.0 |
| p2 | s2 | Yes (needs fact from s1) | Yes | 1.0 |
| p3 | s2 | No | No | 0.0 |
| p4 | s3 | Yes (needs facts from s1, s2) | No | 0.0 |
| p5 | s3 | Yes (needs fact from s1) | Yes | 1.0 |
| p6 | s3 | No (judge-only) | Judge: 0.7 | 0.7 |

```
task_score = (1.0 + 1.0 + 0.0 + 0.0 + 1.0 + 0.7) / 6 = 3.7 / 6 = 0.617

Cross-session probes: p2, p4, p5 → 3 probes, 2 correct
utilization_rate = 2 / 3 = 0.667

Judge contribution: 0.7 / 3.7 = 0.189 (18.9%, within 20% cap)
```

The no-memory control on this same episode would get p2, p4, p5 wrong (facts not
in context) → utilization_rate = 0/3 = 0.0, confirming these probes genuinely
require memory.

## 5. Scorecard — v1 Aggregation

### 5.1 The Original 9-Dim Aggregate (RETIRED for v1)

The original design proposed a 9-dimension weighted aggregate:

```
LHMSB = 0.25×G + 0.20×M + 0.15×D + 0.10×R + 0.10×T + 0.05×R2 + 0.05×E + 0.05×S + 0.05×A
```

This formula is **RETIRED** for v1. v1 implements only 4 dimensions deeply; the
original aggregate assumes all 9 dimensions are scored, which would produce
misleading numbers when 5 dims are unscored. The 9-dim formula remains documented
as a deferred design target for future versions.

### 5.2 v1 Scorecard

The v1 scorecard reports 4 independent metrics, plus the cross-cutting Memory ROI
headline. No single aggregate number is computed from these — each is reported
with its own CI and contextualized by the Memory ROI headline.

**Scorecard table (per system per family per track)**:

| Metric | Type | Units | Reported with |
|---|---|---|---|
| **Memory ROI** | Headline (cross-cutting) | gain per 10³ tokens-equiv | Bootstrap CI, gain floor, Pareto, `undefined-lowcost` flag |
| **Task Score** | Dim 2 | [0, 1] | Bootstrap CI, judge contribution % |
| **Utilization Rate** | Dim 2 | [0, 1] | Conditional on cross-session probes |
| **Drift Index** | Dim 3 | [0, 1] | Per-category breakdown, judge share |
| **Retrieval p@10** | Dim 4 | [0, 1] | Endogenous + oracle modes |
| **Retrieval recall@10** | Dim 4 | [0, 1] | Endogenous + oracle modes |
| **Cost Vector** | Dim 7 (supporting) | tokens-equiv + raw fields | Full vector breakdown, not collapsed |

**Native track** is the primary leaderboard. **Controlled track** (pinned internal
model across systems) is reported separately, never mixed. Sensitivity conditions
(`fake_perfect`, `fake_bad`) are reported in a separate calibration section, not
the leaderboard.

**Pareto front**: systems are plotted on (mean_cost, mean_gain) axes, with the
Pareto-optimal frontier highlighted. A system that achieves tiny gain with tiny
cost cannot "win" on ROI scalar alone if dominated by a system with larger gain
and acceptable cost.

### 5.3 Extensibility

Dimensions 1 (Memory Evolution), 5 (Temporal Reasoning), 6 (Robustness),
8 (Scalability), and 9 (Abstraction) are documented as extension points.
When a future version implements a new dimension, its metric is added to
the scorecard as an independent row; the 9-dim aggregate may be reinstated
once all dimensions are scored. The v1 scorecard design is deliberately
flat (no weighted sum) to avoid the distortion of aggregating scored and
unscored dimensions.
