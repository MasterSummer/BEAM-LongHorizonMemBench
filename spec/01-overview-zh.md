# 01 — 概述、范围、贡献与术语表

## 问题陈述

现代 AI 智能体在长时间跨度上运作：个人助手跨月跟踪用户偏好，科研智能体同化并更新不断增长的证据库，软件开发智能体在需求演进中维护和修订多文件代码库。记忆管理系统旨在为这些智能体提供超出上下文窗口的持久、可演进的回忆能力。然而，该领域缺乏一个标准化的基准来回答部署此类系统时的核心问题：**该记忆系统是否以值得其全部成本的方式提升了任务性能？**

现有基准在部分维度上呈碎片化分布。检索质量（维度4）和时间推理（维度5）已有充分覆盖且趋于饱和（调研了22个基准）。Token/资源效率（维度7）和可扩展性（维度8）在2026年正迅速商品化，相关指标包括 LongMemEval-V2 LAFS、LongMemCode $/query 和 Cost-Accuracy-LTM CpSQ/AES/Pareto。没有一个现有基准能够测量：(a) 记忆系统是否使智能体在长会话中保持稳定的目标导向行为，(b) 使用记忆系统的全生命周期成本（而非仅查询成本），或 (c) 在环境通过撤回和需求变更而演进的过程中，记忆系统在程序化智能体任务上的表现如何。

LongHorizonMemSysBench (LHMSB) v1 通过引入 **Memory ROI** 来弥补这些缺口——这是一个跨维度的核心指标，将性能增益按总记忆系统成本归一化，通过反事实回放协议在两个程序生成的智能体任务族上进行测量。

## v1 范围：四维核心

v1 实现了 LHMSB 跨四个维度的新颖、可防御的核心，以 Memory ROI 为统一的核心指标：

| 维度                   | v1 状态       | 描述                                               |
| -------------------- | ----------- | ------------------------------------------------ |
| **维度2 — 目标导向利用**     | **已实现**     | 智能体是否利用记忆来提升任务完成度？通过程序化任务完成评分标准 + 稀疏判定器进行评分。     |
| **维度3 — 目标漂移与行为稳定性** | **已实现**     | 智能体行为是否跨会话保持稳定？通过对齐探针上的程序化不变量进行测量：`drift_index`。 |
| **维度4 — 检索质量**       | **已实现（辅助）** | 内生（智能体发起）和 oracle（固定查询）检索指标。为其他维度提供支撑，但非核心指标。    |
| **维度7 — Token/资源效率** | **已实现**     | 全生命周期成本仪表化，构成 Memory ROI 的成本分母。                  |

**Memory ROI** 横跨所有四个维度：`ROI = mean(normalized_gain) / mean(normalized_cost)`，附带 bootstrap 置信区间和帕累托分析。

### 扩展点（延迟维度）

以下维度推迟到未来版本。本文档记录了它们的扩展点，以便后续工作可以在不进行架构重新设计的情况下接入。它们明确不属于 v1 评估记分卡或核心指标：

| 维度 | 延迟状态 | 扩展点 |
|------|---------|--------|
| **维度1 — 记忆演进** | 推迟至 v2 | 适配器 `ReflectionCapability` mixin 和 `apply_decay` 钩子已存在；可添加演进评分模块。 |
| **维度5 — 时间推理** | 推迟至 v2 | 类型合约中已存在显式时间查询的探针类型。在先验研究中已饱和——v2 可能作为次要轴添加。 |
| **维度6 — 鲁棒性** | 推迟至 v2 | 对抗性探针生成钩子在模拟器核心中已提供存根。 |
| **维度8 — 可扩展性** | 推迟至 v2 | 数据集大小是可配置参数；扩展实验是独立的 v2 研究。 |
| **维度9 — 抽象** | 推迟至 v2 | 概念汇总探针类型已在模式中定义；评分模块已延迟。 |

v1 并不声称是"面向所有记忆能力的完整基准"。它是一个四维的新颖核心，具有可防御的核心指标和干净的架构，以便日后扩展至全部九个维度。

## 贡献与定位

### 三项新颖性声明

LHMSB v1 在先验研究领域（调研了22个基准；下文引用了6个最接近的竞争者）提出三项新颖性声明：

1. **全生命周期 Memory ROI。** 所有先前的效率感知基准都狭隘地围绕检索/查询 token 定义成本（LongMemEval-V2 LAFS）或每查询美元成本（LongMemCode $/query、Cost-Accuracy-LTM 的 CpSQ）。LHMSB 是第一个在记忆系统 *整个生命周期* 中定义成本的基准：摄取、索引、存储、检索、更新和反思。成本向量捕获智能体 token、记忆系统内部 LLM token、嵌入 token 和调用、存储字节数、检索延迟以及写入/更新/反思延迟。标量化为 token 当量使用一个已声明、固定的转换表，使分母可审计且可复现。分子将增益相对于无记忆反事实进行归一化，产生一个单一的 ROI 数值来回答"为这个记忆系统付费是否值得，以及值得多少？"

2. **智能体任务中标准化的跨系统目标漂移。** 现有基准测量检索漂移（存储的事实是否发生变化？）或嵌入衰减。LHMSB 是第一个以标准化方式操作化 *行为性* 目标漂移的基准，使用程序化不变量：(a) 使用已撤回或已替代的事实，(b) 在无替代事件的情况下违反仍然有效的约束，(c) 无触发事件的行为反转。`drift_index` 是在对齐探针点上的加权违规率，在所有条件下完全相同，因此可以在不同记忆系统之间直接比较漂移。稀疏 LLM 判定器（`lordx64/Qwable-v1`，按修订哈希固定）仅处理程序化无法判定的情况。

3. **程序化智能体任务完成模拟器。** 该领域的主导范式是对话式问答：智能体回答关于合成传记或文档语料库的孤立问题（AMA-Bench、STALE、MemoryAgentBench）。LHMSB 引入了 *程序化任务完成模拟器*，智能体必须在演进中的证据世界中跨多个会话完成一个目标。v1 包含两个任务族：
   - **研究项目**：在合成证据世界上进行自主研究任务，包含撤回（后来被替代的事实）。检查器将每个声明映射到一个合成事实，因此评分是程序化和确定性的。
   - **软件开发**：带有隐藏需求和测试套件的演进规格。任务是一个微小的合成 Python 包，通过确定性 `pytest` 评分；会话期间不允许网络访问或包安装。

   两个任务族共享一个通用的模拟器核心。证据世界是 *固定和外生的*：相同的 `world_event_hash` 对每个 (episode_id, seed) 在所有条件下相同。在 v1 中，智能体动作不会改变世界，从而实现干净的反事实比较。

| 竞争者                   | arXiv / 参考 | 测量内容                                 | LHMSB v1 区分                                                                                 |
| --------------------- | ---------- | ------------------------------------ | ------------------------------------------------------------------------------------------- |

## 规范术语表

所有在 LHMSB 规范中使用的术语在此定义。本术语表中出现的任何术语在所有规范文件（`01-overview.md`、`02-metrics.md`、`03-protocol.md`、`04-datasets.md`、`05-systems.md`）中一致使用。反过来，规范中使用的在 LHMSB 中具有精确定义技术含义的任何术语必须出现在本术语表中。

### 核心实体

**episode（回合）**
一个任务族实例的单次完整运行。一个 episode 包含一个固定的、有序的世界事件序列（注入、变更、撤回）和对齐的探针点，全部从种子派生。Episode 是反事实回放的基本单位：同一个 episode（由 `episode_id` 和 `seed` 标识）在每个记忆条件下运行，事件和探针完全相同。另见：`world_event_hash`、`session`。

**session（会话）**
Episode 中的一个连续交互窗口。智能体感知证据，执行动作，并可能查询或更新其记忆系统。在会话之间，上下文被清除（只有记忆系统持久存在，如果存在的话）。无记忆对照组在会话间是无状态的；所有其他条件下通过其适配器持久化。会话模拟长周期任务中的时间间隔（例如，"明天继续这项研究"）。

**probe（探针）**
在 episode 计划的特定步骤插入的固定测量点。每个探针有一个 `kind`（如 task-completion、drift-check、retrieval）、一个给智能体的 `query`，以及 `gold`（从该步骤的世界状态派生的真实答案 = 已揭示减去已撤回的事实）。探针是 *对齐的*：同一个探针在每个条件下以相同的步骤出现，从而实现配对比较。

**condition（实验条件）**
运行一个 episode 时所使用的特定记忆系统配置。v1 中的排行榜条件包括：`no_memory`、`chromadb`、`mem0`、`letta`、`graphiti`、`cognee`。两个敏感性条件（`fake_perfect`、`fake_bad`）包含在内用于指标验证，但不在排行榜上可见。每个条件配对不同 episode，从而实现反事实比较。

**world_event_hash（世界事件哈希）**
一个稳定的、确定性的哈希值，基于有序的外生事件+探针计划计算。相同的 `world_event_hash` 对每个 (episode_id, seed) 在所有条件下保证每个条件看到完全相同的证据世界。事件计划的变化（例如添加或删除撤回事件）会产生不同的哈希值。此哈希是主要的可复现性和完整性检查。

### Track（赛道）

**native track（原生赛道）**
主要的排行榜赛道。每个记忆系统按原样部署：使用自己的默认配置、自己的内部 LLM（如果有）、自己的索引策略。所有内部 LLM token、嵌入调用、存储字节和检索延迟都被仪表化并计入系统的成本向量。原生赛道回答："当我安装这个记忆系统并运行时，ROI 是多少？"此赛道单独报告，绝不与受控赛道混合。

**controlled track（受控赛道）**
一个次要的子研究赛道，与原生赛道分开报告，绝不合并到主要排行榜中。在受控赛道中，如果记忆系统支持固定其内部模型，则所有系统使用相同的模型（通常与智能体模型相同）。这隔离了记忆架构的效果与内部模型质量的效果。并非所有系统都支持此功能（它是可选的）；受控赛道始终是次要分析。

### 成本与测量

**cost vector（成本向量）**
在给定条件下运行一个 episode 期间产生的所有资源消耗的结构化记录。字段：

| 字段 | 描述 |
|------|------|
| `agent_input_tokens` | 馈入智能体模型的 token（上下文 + 工具结果） |
| `agent_output_tokens` | 智能体模型生成的 token |
| `mem_internal_in_tokens` | 记忆系统内部 LLM 调用消耗的输入 token（添加、搜索、反思、总结等） |
| `mem_internal_out_tokens` | 记忆系统内部 LLM 调用生成的输出 token |
| `embedding_tokens` | 生成嵌入所消耗的 token（文本 → 向量） |
| `embedding_calls` | 嵌入 API 调用次数 |
| `storage_bytes` | 记忆后端存储/消耗的字节数 |
| `retrieval_latency_ms` | 检索/搜索操作中消耗的总挂钟毫秒数 |
| `write_latency_ms` | 写入/插入/更新操作中消耗的总挂钟毫秒数 |
| `reflection_tokens` | 反思/总结/整合调用消耗的 token（`mem_internal_*` 的子集，但为了分析单独跟踪） |
| `num_retrieval_calls` | 搜索/检索操作的总次数 |

成本向量使用已声明、固定的转换表（`configs/cost_weights.yaml`）标量化为 **token 当量**。延迟和存储通过该表转换为 token 当量。美元成本是使用固定价格表的次要、单独报告的列。一次性数据集生成成本和判定器成本被排除在系统成本向量之外，除非系统自身触发了这些成本。
